# engine-native

`engine-native` contains the first native proxy execution engine adapter contracts.

The crate implements `ProxyEngineService` with stable descriptor, structured listener/node/route graph validation, lifecycle, status, and event diagnostics for the in-process native engine. It also exposes the first runtime handle contract for config-driven runtime assembly planning, real loopback TCP listener binding and release, a controlled loopback TCP accept loop, service-owned runtime state, accepted connection counting, relayed connection counting, pre-protocol connection close diagnostics, SOCKS5 greeting version/auth-method read diagnostics, SOCKS5 no-auth method selection and unsupported-auth rejection diagnostics, SOCKS5 auth method response write diagnostics, SOCKS5 command header read and unsupported-command rejection diagnostics, SOCKS5 CONNECT target address read diagnostics, an injectable SOCKS5 CONNECT rich MITM plugin hook, browser capture CONNECT proof token diagnostics, plugin `Reject` application as SOCKS5 CONNECT failure, SOCKS5 route/outbound behavior selection diagnostics, SOCKS5 outbound CONNECT request frame generation diagnostics, SOCKS5 outbound TCP connection plan diagnostics, SOCKS5 outbound TCP connection attempt diagnostics, SOCKS5 outbound CONNECT request write diagnostics, SOCKS5 outbound CONNECT response read diagnostics, SOCKS5 outbound CONNECT response decision diagnostics, SOCKS5 outbound CONNECT relay readiness diagnostics, SOCKS5 outbound CONNECT data relay plan diagnostics, SOCKS5 outbound CONNECT data relay execution diagnostics, SOCKS5 outbound CONNECT client success response readiness diagnostics, SOCKS5 outbound CONNECT client success response write plan diagnostics, SOCKS5 outbound CONNECT client success response write diagnostics, accept-loop SOCKS5 client success response and finite data relay wiring diagnostics, explicit HTTP proxy live plain HTTP request/response rewrite diagnostics, explicit HTTP CONNECT tunnel foundation diagnostics, bounded TLS ClientHello/SNI observation diagnostics, `NativeTlsMitmFoundationReport`, `NativeTlsClientHelloObservationReport`, `NativeControlledTlsTerminationPlanReport`, `NativeHttpsRequestRewritePreviewReport`, caller-provided HTTPS request rewrite preview diagnostics, service start readiness gate diagnostics, SOCKS5 route/outbound unwired rejection diagnostics, SOCKS5 CONNECT failure response write diagnostics, SOCKS outbound handler handoff, startup failure release reports, native runtime events, and foreground lifecycle handoff status. `networkcore-linux start` now wires this service through the current-process foreground lifecycle path while daemon/control-socket operations remain unavailable.

`assess_native_proxy_engine_start_readiness` is the current service-level gate: it records `engine.native.start.runtime_assembly_ready` when a valid configuration can produce a native runtime assembly plan and returns `Ready`. `NativeProxyEngineService::start` then owns the loopback TCP accept loop runtime across `start`/`status`/`events`/`stop`, returning `engine.native.start.running`, `engine.native.runtime.foreground_handoff_ready`, and `engine.native.runtime.accept_loop_ready` when the in-process runtime is active.

The accept loop only accepts loopback TCP connections, reads the first SOCKS5 greeting version/auth-method bytes when present, selects no-auth or rejects unsupported auth methods for diagnostics, writes the SOCKS5 method response, reads the SOCKS5 command header after no-auth selection, reports unsupported commands, and reads CONNECT IPv4/domain/IPv6 target addresses. When a `NativeHttpMitmPluginHook` is injected, the loop maps the CONNECT target to a `HttpMitmEvent`, calls the domain `MitmPluginService`, records the `HttpMitmOutcome`, emits `engine.native.runtime.http_mitm_connect_browser_proof_observed` with a default browser proof token derived from CONNECT authority plus local socks5 proxy URL, and applies `Reject` by writing a SOCKS5 CONNECT general failure response before route/outbound selection. Non-rejected CONNECT requests still select the configured SOCKS outbound handler, generate the upstream SOCKS5 CONNECT request frame diagnostics in memory, create an in-memory outbound TCP connection plan, attempt a bounded TCP connection to IP-literal SOCKS outbound endpoints, write the upstream SOCKS5 CONNECT request when the outbound TCP stream is available, read the upstream SOCKS5 CONNECT response when the request write succeeds, record the outbound response decision, record relay/data-relay/client-success/write-plan readiness, write a SOCKS5 client success response after upstream acceptance, and relay finite client/outbound TCP streams in both directions. `ListenerKind::Http` explicit proxy requests can also parse `http://` HTTP/1.x requests for request/response rewrite and can establish a standard `HTTP/1.1 200 Connection Established` pass-through tunnel for `CONNECT host:443` through the configured SOCKS outbound; before relay it can bounded-peek TLS ClientHello bytes and report SNI without consuming client bytes. `plan_explicit_http_connect_controlled_tls_termination` uses the CONNECT foundation report, ClientHello observation, and NetworkCore CA certificate/private key PEM readiness to produce a controlled downstream TLS termination plan while keeping live HTTPS decryption disabled. `plan_and_apply_https_request_rewrite_preview` can apply reject, redirect, and request header mutation to caller-provided request-phase `https://` input once that controlled termination plan is ready, while deferring body mutation, HTTPS response rewrite, and JavaScript script dispatch. `plan_socks5_connect_http_mitm` remains a plan report with `applied=false`; the accept loop is the only current native site that applies the CONNECT-level `Reject`. Rejected, invalid, failed, or incomplete outbound paths still report route/outbound as unwired, write a SOCKS5 CONNECT general failure response or HTTP 502 response to the accepted client, and then close the accepted and outbound connections. It does not implement BIND/UDP ASSOCIATE execution, long-lived production relay scheduling, UDP, TUN, DNS, live TLS interception, live HTTPS request/response rewrite after CONNECT, HTTPS response rewrite, script mutation application, daemon control, platform mutation, background status, cross-process stop, or service installation.

Verification for this crate is performed only by GitHub Actions, following the repository CI/CD policy.
