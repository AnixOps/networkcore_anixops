# engine-native

`engine-native` contains the first native proxy execution engine adapter contracts.

The crate implements `ProxyEngineService` with stable descriptor, structured listener/node/route graph validation, lifecycle, status, and event diagnostics for the in-process native engine. It also exposes the first runtime handle contract for config-driven runtime assembly planning, real loopback TCP listener binding and release, a controlled loopback TCP accept loop, service-owned runtime state, accepted connection counting, relayed connection counting, pre-protocol connection close diagnostics, SOCKS5 greeting version/auth-method read diagnostics, SOCKS5 no-auth method selection and unsupported-auth rejection diagnostics, SOCKS5 auth method response write diagnostics, SOCKS5 command header read and unsupported-command rejection diagnostics, SOCKS5 CONNECT target address read diagnostics, SOCKS5 route/outbound behavior selection diagnostics, SOCKS5 outbound CONNECT request frame generation diagnostics, SOCKS5 outbound TCP connection plan diagnostics, SOCKS5 outbound TCP connection attempt diagnostics, SOCKS5 outbound CONNECT request write diagnostics, SOCKS5 outbound CONNECT response read diagnostics, SOCKS5 outbound CONNECT response decision diagnostics, SOCKS5 outbound CONNECT relay readiness diagnostics, SOCKS5 outbound CONNECT data relay plan diagnostics, SOCKS5 outbound CONNECT data relay execution diagnostics, SOCKS5 outbound CONNECT client success response readiness diagnostics, SOCKS5 outbound CONNECT client success response write plan diagnostics, SOCKS5 outbound CONNECT client success response write diagnostics, accept-loop SOCKS5 client success response and finite data relay wiring diagnostics, service start readiness gate diagnostics, SOCKS5 route/outbound unwired rejection diagnostics, SOCKS5 CONNECT failure response write diagnostics, SOCKS outbound handler handoff, startup failure release reports, native runtime events, and foreground lifecycle handoff status. `networkcore-linux start` is still not wired to this service-level runtime contract.

`assess_native_proxy_engine_start_readiness` is the current service-level gate: it records `engine.native.start.runtime_assembly_ready` when a valid configuration can produce a native runtime assembly plan and returns `Ready`. `NativeProxyEngineService::start` then owns the loopback TCP accept loop runtime across `start`/`status`/`events`/`stop`, returning `engine.native.start.running`, `engine.native.runtime.foreground_handoff_ready`, and `engine.native.runtime.accept_loop_ready` when the in-process runtime is active.

The accept loop only accepts loopback TCP connections, reads the first SOCKS5 greeting version/auth-method bytes when present, selects no-auth or rejects unsupported auth methods for diagnostics, writes the SOCKS5 method response, reads the SOCKS5 command header after no-auth selection, reports unsupported commands, reads CONNECT IPv4/domain/IPv6 target addresses, selects the configured SOCKS outbound handler for the CONNECT target, generates the upstream SOCKS5 CONNECT request frame diagnostics in memory, creates an in-memory outbound TCP connection plan, attempts a bounded TCP connection to IP-literal SOCKS outbound endpoints, writes the upstream SOCKS5 CONNECT request when the outbound TCP stream is available, reads the upstream SOCKS5 CONNECT response when the request write succeeds, records the outbound response decision, records relay/data-relay/client-success/write-plan readiness, writes a SOCKS5 client success response after upstream acceptance, and relays finite client/outbound TCP streams in both directions. Rejected, invalid, failed, or incomplete outbound paths still report route/outbound as unwired, write a SOCKS5 CONNECT general failure response to the accepted client, and then close the accepted and outbound connections. It does not implement BIND/UDP ASSOCIATE execution, long-lived production relay scheduling, UDP, TUN, DNS, MITM, daemon control, platform mutation, or `networkcore-linux start` wiring.

Verification for this crate is performed only by GitHub Actions, following the repository CI/CD policy.
