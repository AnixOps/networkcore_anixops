# control-runtime

`control-runtime` contains the first pure orchestration use cases for the unified network control kernel.

The crate depends only on `control-domain`. It composes configuration, platform capability, proxy engine, subscription, and MITM plugin ports to prepare configuration, apply explicit inline subscription catalog nodes to `RuntimeConfigRequest.nodes`, start, reload, stop, inspect status, read events, and evaluate initial MITM plugin gates including platform MITM availability, certificate trust denial states, certificate diagnostic preservation on denial, remote script execution denial and unknown states, remote script diagnostic preservation on denial, manifest validation, manifest error denial audit boundaries, manifest error precedence over permission denial, manifest error platform diagnostic preservation on denial, manifest error certificate diagnostic preservation on denial, manifest error denial diagnostic ordering, manifest non-error diagnostic aggregation, manifest diagnostic preservation on permission denial, permission denial diagnostic ordering, plugin result diagnostic aggregation, platform diagnostic aggregation, platform denial diagnostic preservation, permission denial audit boundaries, audit event aggregation, and plugin port error propagation without depending on platform SDKs, external proxy binaries, UI frameworks, network transports, default subscription files, remote subscription fetching, system DNS/TUN mutation, daemon/control sockets, or local filesystem configuration.

The subscription catalog runtime gate uses explicit `SubscriptionService` and `SubscriptionSource` inputs, rejects duplicate node ids with `runtime.subscription.node_id_duplicate`, reports deferred subscription rules with `runtime.subscription.rules_deferred`, and keeps subscription rules out of `ConfigSnapshot.policies` until policy routing is wired.

Verification for this crate is performed only by GitHub Actions, following the repository CI/CD policy.
